# Changelog

All notable changes to LanTransfer are documented here. The format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project aims
to follow [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

This release closes the biggest credibility gaps in the original reference
implementation — cross-platform interop was *asserted* but never verified, file
chunks paid a double-base64 tax, and a malicious relay could MITM the handshake
undetected — and adds a real, server-free browser demo.

### Added

- **Browser web client (`web/`).** A React client that reuses the desktop's
  runtime-agnostic `crypto.ts` / `peer.ts` / `protocol.ts` / `relayClient.ts`
  **verbatim**, lighting up the `Platform='web'` value that already existed in
  the protocol. Builds to a standalone static bundle (relative asset paths).
- **Live, zero-backend demo.** With `?demo=1`, signaling runs over a
  `BroadcastChannel` between browser tabs, so two tabs pair and transfer a
  **real file through the real ECDH + AES-256-GCM + chunking path** with no
  relay at all. Deployed to GitHub Pages via a new workflow.
- **Animated transfer visualization.** A packet-flow lane that fills as chunks
  stream, a live MB/s sparkline (dependency-free inline SVG), an encryption
  shield that flips to “verified” when the SAS is known, and a checksum-verified
  completion state.
- **SAS key-fingerprint verification.** After the ECDH handshake each peer
  derives a Short Authentication String (4 emoji + a 6-digit code) from a hash
  of both public keys. A key-swapping relay makes the codes diverge, so an
  out-of-band comparison detects the man-in-the-middle the handshake alone
  cannot. Surfaced in both the desktop and web UIs, mirrored in the Dart client.
- **Cross-platform interop test suite.** Pinned vectors in
  `shared/interop/interop-vectors.json` (generated by WebCrypto) are consumed by
  **both** a Vitest suite and a Dart test: each imports the SPKI public keys,
  derives the mirror session key, and `open()`s the sealed frame to the exact
  plaintext — verifying byte-level desktop↔mobile compatibility on CI instead of
  asserting it.
- **Relay hardening & observability.** Per-IP connection caps, a per-connection
  token-bucket message-rate limit, a max inbound message size, and a
  max-peers-per-room cap — all configurable via flags/env. A new `/metrics`
  endpoint exposes rooms, peers, and messages-routed in Prometheus format.
- **Docker Compose one-command run** with an optional Caddy TLS front (the `tls`
  profile) so a public `wss://` endpoint comes up with automatic HTTPS.
- **Test coverage** via `@vitest/coverage-v8` (desktop shared core at ~87%
  lines) plus new tests for the relay client, the binary frame codec, the SAS,
  the loopback transport, and the relay rate-limiter/room-cap/metrics.

### Changed

- **Binary data-channel frames (protocol v2).** File chunks now travel as a
  compact binary frame — `magic | flags | seq(u32) | transferId(16) | bytes`,
  AES-GCM-sealed and sent as an `ArrayBuffer` — instead of base64-in-JSON
  strings sent twice through base64. This removes ~2× base64 inflation and a
  JSON encode/decode per chunk on the throughput-critical path. Control messages
  (offer/accept/complete/cancel) stay JSON. `PROTOCOL_VERSION` bumped to `2`
  across the desktop, mobile, and relay.
- README overhauled with a badge row, a clearer architecture blurb, the demo
  link, and an honest, expanded security model (including the MITM threat and
  the SAS mitigation).

### Security

- Documented the active-attacker (malicious relay) threat model and added the
  SAS mitigation, rather than leaving the gap unacknowledged.
- The relay can no longer be trivially abused by a single client: connection,
  message-rate, message-size, and room-occupancy limits are enforced.

## [0.1.0] — 2026

Initial reference implementation: an Electron desktop app, a Flutter mobile
client, and a self-hostable Go signaling relay sharing one wire protocol.
End-to-end encryption (ephemeral ECDH P-256 → HKDF-SHA-256 → AES-256-GCM),
chunked transfers with SHA-256 verification, and an in-memory end-to-end
transfer test on the desktop.
